The shift

Before

You prompt. The agent responds. You inspect. You prompt again.

Now

A system discovers work, delegates it, checks it, persists state, and decides what comes next.

The human moves out of the inner loop and stays accountable at the outer loop.

The trivial loop


while True:
    response = model(context)
    if response.has_tool_calls():
        results = run_tools(response.tool_calls)
        context += results
    else:
        break

Model → tools → context → repeat. Nobody is competing on the while statement.

So what are we engineering?

Our job is not the while loop. Our job is the harness, the verifier, the memory, and the brakes.

Working definition

Loop engineering is designing a recurring agent system that discovers, delegates, verifies, persists, decides, and stops.

A cron job repeats. A loop decides.

Loop lifecycle

Trigger→ Discover→ Delegate→ Act→ Verify→ Persist→ Decide

The load-bearing step is verification.

What makes it a loop?

Heartbeat

Schedule, event, webhook, failing check, or adaptive cadence.

Memory

Progress lives outside the conversation: files, issues, boards, traces.

Decision

Continue, retry, stop, or escalate for a reason.

Maker vs checker

Maker

Proposes the smallest change that could satisfy the goal.

Checker

Defaults to reject, verifies with evidence, and decides if done is real.

The worker does not grade its own homework.

/loop vs /goal

PrimitiveWhat it doesRisk
/loopRepeats on cadenceCan repeat without proving success

Hard part 1: stopping

An agent ending its turn is not the same as completing the task.

"Done" should mean tests pass, not the agent feels good.

Honest stop conditions

Goal met

The verifier confirms the done-condition.

Budget spent

Iterations, time, tokens, or dollars hit the cap.

Stalled

Same failure repeats with no new evidence.

Needs human

Risky, ambiguous, taste-driven, or regulated.

Hard part 2: context rot

Long loops accumulate stale tool output, dead ends, and obsolete reasoning.

A rotted context makes worse decisions, which add more noise, which rots the context further.

Context hygiene

  • Compact long runs into a clean handoff.
  • Offload huge outputs to files; feed back only relevant slices.
  • Use sub-agents for messy subtasks; return distilled results.
  • Govern durable memory like code.

Context is a budget, not a bucket.

Hard part 3: tool design

Few, focused tools

If a human cannot tell which tool to use, the agent has no chance.

Idempotent writes

Retries must not create duplicate customers, tickets, commits, or charges.

Agent-readable errors

An error should tell the next move.

Least privilege

Scope by what the loop can break, not what you hope it does.

Good error messages become prompts


Bad:
Migration failed.

Good:
Migration failed because column user_id is nullable in 3 rows.
Next step: backfill those rows or add a nullable migration first.

Hard part 4: something says no

A loop with no critic is just an agent nodding along to its own work.

Use tests, typechecks, linters, rubrics, reviewers, and human gates.

What loops are good for

  • CI failure repair
  • Dependency upgrades
  • Flaky-test hunts
  • Doc drift repair
  • Issue triage
  • PR babysitting
  • Benchmark tuning
  • Alert triage

Recurring + bounded + machine-checkable.

When not to loop

  • No machine-checkable done-condition.
  • One-off task that will not amortize loop design.
  • Taste, strategy, architecture, or product judgment.
  • High-intent-debt legacy work.
  • Safety-critical or regulated output with no human gate.

Security: the lethal trifecta

Private data

Repos, secrets, tickets, customer records.

Untrusted input

Issues, PRs, web pages, logs, dependency text.

Outbound channel

PRs, Slack, email, web requests, artifacts.
