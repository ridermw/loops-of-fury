The one-sentence pitch: this repo doesn't just teach loop engineering — it runs a real loop on itself. The slides you're looking at are maintained by the very pattern they describe. The thesis

What if the deck improved itself?

A control plane reads the decks, makes one small change, proves it is better, and ships to main — then does it again. Over an 8-hour window, a hundred times.

This is the whole idea in one question. A normal deck is frozen the moment you stop editing it. This one has a control plane that wakes up, reads its own slides, makes exactly one small change, runs an objective check to prove the change is actually better, and — only if it passes — commits and pushes to main. Then it does it again.

The numbers matter: an eight-hour budget, around a hundred iterations, and zero pull requests. No review queue, no human approving each diff. The discipline that makes that safe is the rest of this deck. The thesis the repo teaches — done-condition, narrow maker, independent checker, explicit contract — is exactly the machine improving this slide right now.

Four roles, one loop

Maker

Copilot CLI, narrow by design. Edits exactly one deck, no shell. Proposes a change; never decides it is done.

Driver

Owns git. Snapshots, commits, pushes, reverts. The only thing in the system that touches main.

Checker

The objective gate. Renders the deck in real Chromium and returns RED or GREEN with evidence.

Brain

Pure decisions, no side effects: continue, retry, switch axis, or stop.

This is separation of powers, and it's the core mental model. The maker is Copilot CLI given only two tools — view and write — and allowed to touch only the decks. It proposes a change and nothing more: it can't run shell, can't see the checker, can't decide it's done. The driver (driver.mjs) owns every git operation — snapshot, commit, push, revert — and is the only thing that touches main.

The checker (check.mjs plus anchors, hygiene, and visual) re-renders the edited deck in a real Chromium and returns RED or GREEN with evidence — never an opinion. The brain (brain.mjs) is pure decision logic with an injected clock: given an outcome it says continue, retry, switch axis, or stop. The single most important rule: the maker proposes, but it can never mark its own work as done, and it can never edit the machine that judges it.

One iteration

Pick weakest axis→ Maker edits one deck→ Diff-gate→ Secret-scan→ Manifest→ Render + floors→ Commit→ Push to main→ Live verify→ Ledger→ Decide

RED at any gate → the whole iteration reverts byte-for-byte. GREEN all the way → it is already on main.

Walk the pipeline left to right — this is one iteration. First pick the weakest axis from the scoreboard. The maker edits one deck. Then a stack of gates runs in order: the diff-gate checks that only decks and assets changed; the secret-scan reads the added lines for tokens; the manifest checks that no control-plane file drifted; then the deck is rendered and the objective floors run. Only if all of that is green does the driver commit, push to main, and verify the live Pages deploy. Finally it records the win in the ledger and the brain decides what's next.

The key property is all-or-nothing. RED at any gate reverts the whole iteration byte-for-byte — the working tree goes back to the last good state as if nothing happened. GREEN all the way through means the change is already live on main before the next iteration starts. There is no “mostly done” state and no half-committed deck.

The gate is the only authority

No human approves a change. No pull request waits in a queue.

RED

The deck is reverted to the last good state. The maker retries or the loop switches axis.

GREEN

Render, structure, links, and visuals all pass — so the commit has already shipped.

The checker is .loop/check.mjs — never npm test, never a reviewer.

This is the part that makes people nervous, so address it head-on. Nothing here waits for a human. There is no pull request, no reviewer, no approval step. The thing that decides whether a change reaches main is the objective gate, and only the objective gate.

RED means the render failed, or a slide disappeared, or an internal link broke, or the layout overflowed — and the deck is restored to its last good state automatically. GREEN means render, structure, links, and visuals all passed, so the commit has already shipped. Critically, the checker is .loop/check.mjs — a purpose-built deck gate — not npm test. The repo's test suite is intentionally red; it's the workshop exercise. Wiring the loop to npm test would be the classic mistake: checking the wrong done-condition.

The loop cannot edit its own gate

Diff-gate

Maker may touch decks and assets only. Anything else in the diff → revert.

Secret-scan

The staged diff is scanned for tokens. A match → revert, redacted in the log.

Control-manifest

Every .loop file is SHA-256 verified each iteration. Drift → escalate.

Pre-push hook

A second, out-of-process barrier on the path to main.

Live-Pages verify

Re-render the deployed SHA. Genuinely broken → forward-revert.

Auto-revert

Any failure restores the last good deck. main never ships broken.

Defense in depth — an edit has to clear all six of these to reach main. The diff-gate runs before the checker and reverts if anything outside the deck-and-asset allowlist changed. The secret-scan reads the added lines for credential-shaped tokens and redacts any match in the logs. The control-manifest is the keystone: a SHA-256 hash of every control-plane file, verified every single iteration.

That manifest is why the maker can't cheat. The maker can write decks and assets, but the entire .loop machine — the gate, the brain, the config — is hash-checked, so changing the gate is drift and gets caught instantly. On top of that there's an out-of-process pre-push hook (a completely separate code path to main), a live-Pages re-render of the deployed SHA, and auto-revert on any failure. Least privilege, an immutable control plane, and two independent barriers: the loop physically cannot weaken its own checker.

Four axes, one floor

  • Render — the deck builds and shows every slide.
  • Hygiene — internal anchors and assets resolve.
  • Freshness — external citations stay alive.
  • Delight — an LLM judge scores it against the deck's own design system.

The objective floor

render + anchors + hygiene-hard + visual. The maker can never drift below it.

Delight only chooses what to work on next. It can inform, but it can never revert main.

Each iteration works exactly one axis. Render and hygiene are hard-gated — a broken render or a dead internal link reverts main. Freshness (external citations) is soft: a flaky third-party link never reverts the deck; it's recorded and only escalates after several consecutive misses. Delight is an LLM judge scored against this deck's own design tokens — not generic “make it pretty” — and it is completely non-gating.

This is the most important safety idea in the whole system, called Premise 3: only the objective floor protects main. The floor is render plus content anchors plus the hard hygiene checks plus visual invariants. Subjective quality — the delight judge, pixel drift, link flakiness — can inform which axis to work on next, but it can never revert main. We never let a taste model gate a public branch.

Always work the weakest axis

A scoreboard tracks every axis. Each iteration spends its next move on whichever one is lagging; ties round-robin.


{
  "seq": 110,
  "axes": {
    "render":    { "score": 27 },
    "hygiene":   { "score": 26 },
    "freshness": { "score": 27 },
    "delight":   { "score": 26 }
  }
}

No weighted optimizer. Lowest score wins — a new reader understands it in 30 seconds.

Axis selection is deliberately dumb, and that's a feature. A persistent scoreboard tracks a score per axis, and the next iteration always goes to the lowest-scoring one. A verified green nudges that axis's score up, so attention naturally shifts away from it; a revert or no-op leaves it weak, so the loop comes back to it. Ties break round-robin by least-recently-picked, using a monotonic sequence counter — not wall-clock — so the ordering is exact.

Notice what this isn't: there's no weighted optimizer, no learned policy, no reward function to tune. A new engineer can read scoreboard.json and understand exactly why the loop is about to work hygiene next, in about thirty seconds. When you're running something unattended for eight hours, legibility beats cleverness. If you can't explain why the loop did what it did, you can't trust it.

It files its own paperwork

loop-run

One heartbeat issue per run. Every commit references it with a non-closing Refs: #N. Closed on a clean end; left open and labelled on a failure.

loop-task

You file an issue labelled loop-task. The next run drains the queue, lands it through the same gate, and closes it — or labels it needs-review and moves on.

Observability, not an approval gate. No step ever waits on a human.

Two separate GitHub flows, and neither is ever an approval gate. The loop-run issue is observability: one issue per run, opened before the first commit, and every commit carries a non-closing Refs: #N trailer so you can trace exactly which run produced which change. A clean finish closes it; a failure leaves it open with a loop-escalation label and a diagnosis comment for a human.

The loop-task queue is how you direct the loop. You file an ordinary issue labelled loop-task, and before its normal polish work each run drains those oldest-first through the same gated, live-verified spine. If the ask lands within the gates, it commits, comments the deploying SHA, and labels it loop-done. If it can't — a structural change like adding or removing slides — the maker makes no change, the issue is labelled loop-needs-review, and the loop moves on. It's bounded by a per-run task cap, the time budget, and retry limits.

Crash-safe and bounded

  • Every run stamps a UUID and refreshes a heartbeat.
  • A dead run is finalized — its issue is never inherited.
  • A live run refuses to start a second driver of main.
  • 8-hour hard cap.
  • Stop after K no-op iterations.
  • Retry K times, then switch axis.
  • Too many reverts → pause and escalate.

Unattended means it will crash eventually, so the lifecycle is built for it. Every run stamps a unique UUID into run.json and refreshes a heartbeat each iteration. At startup the loop classifies whatever it finds: the same UUID means resume; a different run whose heartbeat is older than fifteen minutes gets finalized as dead — and crucially its run-issue is never inherited; a different run still heartbeating means refuse to start, because you must never have two drivers pushing main at once.

The terminators are all code, not vibes. An eight-hour hard wall-clock cap. Stop after five consecutive no-op iterations. Retry an axis three times, then switch. And an anti-thrash churn cap: too many forward-reverts inside a thirty-minute window pauses the run and escalates. The loop stops honestly — it never spins forever pretending to make progress.

Proof of work

The deck counts its own wins

A driver-owned ledger slide regenerates from .loop/ledger.json, tallying every verified improvement by axis.

The deck becomes living proof of its own thesis.

This is the part that closes the loop on the thesis. ledger.mjs owns a slide inside index.html, regenerated from ledger.json, that counts every verified improvement broken down by axis. The research deck literally shows a running tally of the loop's own successful commits — the presentation becomes living proof of its own argument.

Two engineering details worth mentioning if asked. It only reuses existing components — big-number, pill, flow — so it adds no new headings or tokens and never trips the anchor floor. And it's CRLF-preserving and idempotent: re-rendering the committed empty seed is a byte-for-byte no-op, so the very first unattended run doesn't produce a spurious “the ledger changed” commit. A self-report that pollutes its own diff would undermine the whole point.

Run it


$env:GH_TOKEN    = "<push token>"
$env:LOOP_MAKER  = "copilot"
$env:LOOP_COMMIT = "1"

npm run loop:preflight   # refuse to start if the gate can't catch a broken deck
npm run loop:run         # drain loop-task issues, then polish the weakest axis

npm run loop:selftest proves the gate end-to-end without touching main.

Show how little it takes to run. Set a push token, point LOOP_MAKER at the real Copilot CLI maker (the default is a no-op maker that just proves the spine without spending credits), and set LOOP_COMMIT=1 to allow pushes. Then loop:preflight does a red-on-broken go/no-go: it deliberately breaks a deck and confirms the gate catches it, and refuses to start if the gate can't. You never run an autonomous loop whose checker you haven't just proven works.

Then loop:run drains any loop-task issues and polishes the weakest axis until a stop condition. The safest thing to demo live is loop:selftest: it injects a known-good deck edit, watches the gate, manifest, and check all go green, then reverts clean — proving the entire pipeline end-to-end without ever touching main. There's a companion loop:selftest-gate that injects a forbidden change and proves the diff-gate blocks it.

Land the real takeaway. The decks are almost incidental — they're just the thing this particular loop happens to improve. What's reusable is the shape: a machine-checkable done-condition, a maker kept deliberately narrow, a checker the maker can't see or weaken, and a loop contract whose stop conditions are code. Point that same shape at CI repair, flaky-test triage, dependency bumps, or doc maintenance and the safety properties carry over unchanged.
